# 需求决策记录

本文档记录在进入关卡验收和评分前必须先确认的产品/机制规格。

## 通关标准

状态：已确认。

这不是 audit 或 scorer 应该自行推断的内容。它和“玩家能拉箱子”“传送门成对工作”一样，是原型规格的一部分。只有先声明通关标准，后续工具才有资格检查关卡是否满足它。

当前 `pull_portal_fallback` 的正式玩家关卡通关标准：

```yaml
win:
  type: player_on_goal
  target: goal
```

含义：玩家需要到达指定目标格。

当前 `pull_portal_fallback` 包里存在两类用途混在一起：

- 机制 witness：为了证明某个规则分支可触发，使用 `event_occurs` 作为临时胜利条件。
- 玩家关卡：应该有面向玩家的统一通关目标，例如玩家到达出口、箱子到目标、或二者组合。

当前约定：

- 箱子目标不是这个原型的默认通关条件。
- `event_occurs` 可以用于机制 witness / test fixture。
- 正式玩家关卡应使用 `player_on_goal`，或在明确设计理由下提交规格变更。

## 评分器边界

状态：未实现。

当前 evaluator 只能证明非常有限的事情：

- 可解性。
- 最短解事件覆盖。
- 部分反事实不可解。
- 有限状态预算内没有找到绕过目标事件的解。

它不能评价：

- 关卡是否有趣。
- 教学是否清晰。
- 目标知识是否在认知上必要。
- 是否只是同一个走路关的平移/拉伸。
- 非形式化战术语义是否真的发生，例如“箱子堵住出口”或“移动传送门打开通路”。

因此，在评分器定义前，不应把 `PASS`、`accepted`、或知识事件覆盖解释为关卡质量通过。

## Agent 规格确认清单

状态：第一版。

后续把能力泛化成 skill/MCP 时，agent 在生成机制、知识和关卡前必须确认或记录以下事项。没有确认时，不应自行假设。

1. 通关标准是什么？
   例如玩家到达指定位置、箱子到目标、收集全部物品、触发某种状态组合，或每关允许不同目标类型。

2. 预期关卡规模是多少？
   包括大概关卡数量、每个机制的教学/应用/挑战比例、以及原型阶段是否只需要少量 witness。

3. 未说明的机制交互如何处理？
   例如传送门被推入另一个传送门、箱子堵在传送门上、多个传送门组重叠、失败动作是否消耗步数。

4. 哪些机制是侧重点？
   需要区分核心机制、辅助机制、只需一笔带过的边界情况，以及不进入本次课程的机制。

5. 每个机制的具体语义是什么？
   例如进入传送门后落在另一扇门所在格、门外一格、还是按进入方向偏移一格；物体是否随玩家传送；移动传送门时是否保留朝向或组关系。

6. 同类对象是否允许多实例？这些实例是否对玩家可区分？
   例如多个普通箱子、彩色箱子、编号钥匙、成组传送门、多个同类角色。若对象不可区分，工具可以用临时实例 id 做证据追踪，但知识和关卡目标不应写成玩家需要识别某个具体 id。

7. 课程是否需要显式覆盖对象数量变化？
   例如单箱、双箱、三箱是否分别构成递进；两个箱子是否只是更多元素，还是要求两个不同实例承担不同因果角色。

8. 哪些内容可以由工具默认化，哪些必须显式确认？
   严谨逻辑解谜里，错误默认值会污染知识列表、课程规划、求解器和关卡质量判断。默认值必须可追溯，并在报告中暴露。
